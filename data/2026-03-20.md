### [1] [Conditional Execution of Transpiler Passes Based on Per-Script Feature Detection](https://arxiv.org/abs/2603.18049)
*Rishipal Singh Bhatia*

Main category: cs.PL

TL;DR: Google Closure Compiler改进：通过动态追踪和维护JavaScript语言特性集合，条件执行转译pass，减少不必要的AST遍历，显著提升编译性能


<details>
  <summary>Details</summary>
Motivation: 传统JavaScript编译器采用基于目标语言级别的固定转译流水线，即使输入代码不包含某些语言特性也会执行相应的转译pass，导致大量计算浪费和性能损失

Method: 在Google Closure Compiler中实现特性集(FeatureSet)动态维护机制，在脚本级别追踪实际存在的语言特性，条件执行转译pass，包括策略性pass排序和动态验证转译代码的正确性

Result: 在大规模生产代码库中评估显示，显著减少了编译时间，节省了计算和内存使用

Conclusion: 通过动态追踪语言特性并条件执行转译pass，可以有效减少JavaScript编译器的不必要计算开销，提升编译性能，该方法已成功应用于生产环境

Abstract: As the ECMAScript specification evolves, industrial-scale JavaScript compilers face the challenge of supporting modern language syntax while maintaining compatibility for diverse execution environments. Traditionally, compilers solve this by running transpilation passes in a monolithic pipeline, where the transpilation passes are chosen to execute strictly based on a target language level. This results in significant computational waste, as compilers perform expensive Abstract Syntax Tree (AST) traversals to lower features that may not exist in the actual input source code. We present a compiler improvement that conditionally executes transpiler passes based on accurately tracking and dynamically maintaining the exact set of language features present in the compilation unit throughout the transpilation process. It is implemented in the production Google Closure Compiler. By populating and maintaining a FeatureSet at every JavaScript script-level, it dynamically skips running unnecessary lowering passes. We detail the architectural safeguards -- including strategic pass ordering and dynamic validation of the transpiled code for feature-correctness. Evaluation of this improvement on large-scale production monorepos produced a considerable reduction in compilation time and saved compute and memory usage.

### [2] [TENSURE: Fuzzing Sparse Tensor Compilers (Registered Report)](https://arxiv.org/abs/2603.18372)
*Kabilan Mahathevan,Yining Zhang,Muhammad Ali Gulzar,Kirshanthan Sundararajah*

Main category: cs.PL

TL;DR: TENSURE：首个针对稀疏张量编译器（STC）的可扩展黑盒模糊测试框架，使用爱因斯坦求和表示法生成复杂张量收缩运算，通过约束生成算法保证100%语义有效性，在TACO和Finch系统中发现大量崩溃和静默编译错误。


<details>
  <summary>Details</summary>
Motivation: 稀疏张量编译器需要从高级声明式规范合成复杂、不规则的控制流，极易出现正确性缺陷。现有测试框架无法充分测试这些编译器的任意循环合成能力，而通用基于语法的模糊测试器难以生成有效输入。

Method: 1) 使用爱因斯坦求和表示法作为通用输入抽象；2) 提出新颖的基于约束的生成算法保证100%语义有效性；3) 引入语义保持的变异算子，利用代数交换性和存储格式异质性进行蜕变测试。

Result: 在TACO和Finch两个最先进系统上的评估显示广泛脆弱性，特别是在TACO中，TENSURE在大多数生成的测试用例中暴露了崩溃或静默编译错误，显著优于基线语法模糊测试器约3.3%的有效率。

Conclusion: 稀疏编译生态系统迫切需要专门的测试工具，TENSURE框架有效暴露了现有稀疏张量编译器中的关键缺陷，证明了针对该领域定制测试方法的重要性。

Abstract: Sparse Tensor Compilers (STCs) have emerged as critical infrastructure for optimizing high-dimensional data analytics and machine learning workloads. The STCs must synthesize complex, irregular control flow for various compressed storage formats directly from high-level declarative specifications, thereby making them highly susceptible to subtle correctness defects. Existing testing frameworks, which rely on mutating computation graphs restricted to a standard vocabulary of operators, fail to exercise the arbitrary loop synthesis capabilities of these compilers. Furthermore, generic grammar-based fuzzers struggle to generate valid inputs due to the strict rules governing how indices are reused across multiple tensors.
  In this paper, we present TENSURE, the first extensible black-box fuzzing framework specifically designed for the testing of STCs. TENSURE leverages Einstein Summation (Einsum) notation as a general input abstraction, enabling the generation of complex, unconventional tensor contractions that expose corner cases in the code-generation phases of STCs. We propose a novel constraint-based generation algorithm that guarantees 100% semantic validity of synthesized kernels, significantly outperforming the ~3.3% validity rate of baseline grammar fuzzers. To enable metamorphic testing without a trusted reference, we introduce a set of semantic-preserving mutation operators that exploit algebraic commutativity and heterogeneity in storage formats. Our evaluation on two state-of-the-art systems, TACO and Finch, reveals widespread fragility, particularly in TACO, where TENSURE exposed crashes or silent miscompilations in a majority of generated test cases. These findings underscore the critical need for specialized testing tools in the sparse compilation ecosystem.

</details>


### [3] [Leveraging Large Language Models for Generalizing Peephole Optimizations](https://arxiv.org/abs/2603.18477)
*Chunhao Liao,Hongxu Xu,Xintong Zhou,Zhenyang Xu,Chengnian Sun*

Main category: cs.PL

TL;DR: LPG使用大型语言模型来自动化窥孔优化的泛化过程，相比现有方法显著提升了泛化能力和成功率。


<details>
  <summary>Details</summary>
Motivation: 窥孔优化是现代编译器优化的核心组件，但将具体优化实例泛化为通用重写规则需要大量手动工作和领域专业知识。现有自动化方法如Hydra存在搜索空间爆炸、泛化不足和对不同指令域支持有限等问题。

Method: LPG采用大型语言模型辅助的闭环工作流程，结合LLM驱动的符号常量泛化、结构泛化、约束松弛和位宽/精度泛化，并通过语法验证、语义验证和收益性检查提供反馈。

Result: 在LLVM生态系统的真实窥孔优化问题上，LPG成功泛化了102个优化中的90个。在与Hydra可比的整数优化子集上，LPG泛化了81个中的74个，而Hydra仅泛化了35个。

Conclusion: LPG框架通过结合LLM的语义抽象和探索推理能力与形式化分析，有效解决了窥孔优化泛化的难题，显著优于现有方法。

Abstract: Peephole optimizations are a core component of modern optimizing compilers. It rewrites specific instruction into semantically equivalent but more efficient forms. In practice, creating a new peephole optimization often starts from a concrete optimization instance and requires lifting it into a more general rewrite rule that matches a wider range of instruction patterns. This generalization step is critical to optimization effectiveness, but it is also difficult: producing rules that are both correct and sufficiently general typically demands substantial manual effort and domain expertise. Existing approaches such as Hydra attempt to automate this task with program synthesis, but their generalization capability is often limited by search-space explosion, under-generalization, and restricted support for diverse instruction domains.
  We present LPG, large language model aided peephole optimization generalization, a framework that uses large language models (LLMs) to generalize peephole optimizations. The design of LPG is motivated by the observation that LLMs are effective at semantic abstraction and exploratory reasoning, while formal analyses are necessary to ensure that generated rules are sound and profitable. Based on this observation, LPG adopts a closed-loop workflow that integrates LLM-driven symbolic constant generalization, structural generalization, constraint relaxation, and bitwidth/precision generalization with feedback from syntactic validation, semantic verification, and profitability checking.
  We evaluate LPG on real-world peephole optimization issues drawn from the LLVM ecosystem. Overall, LPG successfully generalizes 90 out of 102 optimizations. On the integer-focused subset that is directly comparable to Hydra, LPG generalizes 74 out of 81 optimizations, whereas Hydra generalizes 35.

</details>

### [4] [Quine: Realizing LLM Agents as Native POSIX Processes](https://arxiv.org/abs/2603.18030)
*Hao Ke*

Main category: cs.OS

TL;DR: Quine提出将LLM智能体实现为原生POSIX进程的运行时架构，利用操作系统已有的隔离、调度和通信机制，避免应用层重复造轮子。


<details>
  <summary>Details</summary>
Motivation: 当前LLM智能体框架在应用层实现隔离、调度和通信，但这些功能操作系统已经成熟提供。重复实现这些机制效率低下，且无法充分利用OS内核的稳定性和性能优势。

Method: 将LLM智能体映射为POSIX进程：身份对应PID，接口使用标准流和退出状态，状态对应内存、环境变量和文件系统，生命周期使用fork/exec/exit。通过单个可执行文件递归生成自身实例来实现。

Result: Quine继承了内核的隔离、组合和资源控制能力，自然支持递归委托、通过exec进行上下文更新以及与shell原生组合。同时揭示了POSIX进程模型在认知任务中的局限性。

Conclusion: 基于OS进程模型为LLM智能体提供了稳健的执行基础，但指出需要超越进程语义的扩展：任务相对世界和可修改时间。参考实现已在GitHub开源。

Abstract: Current LLM agent frameworks often implement isolation, scheduling, and communication at the application layer, even though these mechanisms are already provided by mature operating systems. Instead of introducing another application-layer orchestrator, this paper presents Quine, a runtime architecture and reference implementation that realizes LLM agents as native POSIX processes. The mapping is explicit: identity is PID, interface is standard streams and exit status, state is memory, environment variables, and filesystem, and lifecycle is fork/exec/exit. A single executable implements this model by recursively spawning fresh instances of itself. By grounding the agent abstraction in the OS process model, Quine inherits isolation, composition, and resource control directly from the kernel, while naturally supporting recursive delegation, context renewal via exec, and shell-native composition. The design also exposes where the POSIX process model stops: processes provide a robust substrate for execution, but not a complete runtime model for cognition. In particular, the analysis points toward two immediate extensions beyond process semantics: task-relative worlds and revisable time. A reference implementation of Quine is publicly available on GitHub.

</details>

### [5] [Don't Vibe Code, Do Skele-Code: Interactive No-Code Notebooks for Subject Matter Experts to Build Lower-Cost Agentic Workflows](https://arxiv.org/abs/2603.18122)
*Sriram Gopalakrishnan*

Main category: cs.AI

TL;DR: Skele-Code是一个面向非技术用户的AI工作流构建工具，采用自然语言和图形界面，支持增量式笔记本开发，将步骤转换为代码，降低多代理系统的token成本。


<details>
  <summary>Details</summary>
Motivation: 为降低非技术用户使用AI代理构建工作流的门槛，减少多代理系统方法中的token成本，提供更模块化、可扩展的工作流开发方案。

Method: 采用自然语言和图形界面，支持增量式笔记本开发，每个步骤转换为包含必要函数和行为的代码。AI代理仅用于代码生成和错误恢复，不参与编排或任务执行，采用代码优先、代理辅助的方法。

Result: Skele-Code能够生成模块化、易于扩展和共享的工作流，这些工作流可作为代理的技能或其他工作流的步骤使用，同时相比多代理系统方法降低了token成本。

Conclusion: Skele-Code通过代码优先、代理辅助的方法，为非技术用户提供了高效、低成本的工作流构建工具，实现了模块化、可扩展的工作流开发。

Abstract: Skele-Code is a natural-language and graph-based interface for building workflows with AI agents, designed especially for less or non-technical users. It supports incremental, interactive notebook-style development, and each step is converted to code with a required set of functions and behavior to enable incremental building of workflows. Agents are invoked only for code generation and error recovery, not orchestration or task execution. This agent-supported, but code-first approach to workflows, along with the context-engineering used in Skele-Code, can help reduce token costs compared to the multi-agent system approach to executing workflows. Skele-Code produces modular, easily extensible, and shareable workflows. The generated workflows can also be used as skills by agents, or as steps in other workflows.

</details>
