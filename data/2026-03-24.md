### [1] [Can LLMs Perform Synthesis?](https://arxiv.org/abs/2603.20264)
*Derek Egolf,Yuhao Zhou,Stavros Tripakis*

Main category: cs.PL

TL;DR: LLMs在程序综合任务上与符号工具对比：在LTL反应式综合、语法引导综合、分布式协议综合和递归函数综合四个领域，符号工具在解决更多基准测试和执行时间方面均优于或与GPT-5相当，显著优于32B参数的Qwen模型。


<details>
  <summary>Details</summary>
Motivation: 研究LLMs与符号工具在程序综合任务上的性能比较，评估当前前沿LLM在程序综合领域的能力边界。

Method: 在四个程序综合领域（LTL反应式综合、语法引导综合、分布式协议综合、递归函数综合）进行对比实验：1）为每个领域选择最先进的符号工具；2）使用32B参数的Qwen LLM配合符号验证器进行迭代求解；3）使用GPT-5进行单次生成并验证输出。

Result: 在所有领域中，符号工具解决的基准测试数量均超过Qwen，且优于或与GPT-5相当。在执行时间方面，符号工具在所有领域都优于GPT-5，且优于或接近Qwen，尽管LLMs运行在更强大的硬件上。

Conclusion: 符号工具在程序综合任务上仍然比当前最先进的LLMs更具优势，特别是在解决能力和执行效率方面，表明符号方法在该领域仍有重要价值。

Abstract: How do LLMs compare with symbolic tools on program synthesis tasks? We investigate this question on several synthesis domains: LTL reactive synthesis, syntax-guided synthesis, distributed protocol synthesis, and recursive function synthesis. For each domain, we choose a state-of-the-art symbolic tool and compare it to an open-source, 32 billion parameter version of the Qwen LLM and the proprietary, frontier LLM GPT-5. We couple Qwen with a symbolic verifier and run it repeatedly until it either produces a solution that passes the verifier, or until there is a timeout, for each benchmark. We run GPT-5 once per benchmark and verify the generated output. In all domains, the symbolic tools solve more benchmarks than Qwen and either outperform or are about on par with GPT-5. In terms of execution time, the symbolic tools outperform GPT-5 in all domains, and either outperform or are very close to Qwen, despite the fact that the LLMs are run on significantly more powerful hardware.

### [2] [Accompanist: A Runtime for Resilient Choreographic Programming](https://arxiv.org/abs/2603.20942)
*Viktor Strate Kløvedal,Dan Plyukhin,Marco Peressotti,Fabrizio Montesi*

Main category: cs.PL

TL;DR: 提出Accompanist，一个为Choral编排编程语言设计的弹性运行时，用于在去中心化环境中实现可靠的saga事务，通过计算重放来屏蔽故障，保持程序简单且无需修改编译器。


<details>
  <summary>Details</summary>
Motivation: 在面向服务架构中，编排器虽然简化了故障恢复和回滚，但由于硬件限制和安全策略并不总是可用。没有中央编排器时，实现弹性的saga事务变得困难。编排编程虽然提供了全局视图，但依赖于网络可靠性和服务正常运行时间的强假设，这些假设在实际中往往不成立。

Method: 提出Accompanist运行时，将编程接口与可定制的运行时协同设计，能够重放计算以屏蔽故障。假设编排程序是确定性的、saga内的事务是幂等的、消息写入持久消息队列。将编译后的代码部署到与现有代码库中服务并行的"sidecars"中。

Result: 提出了一个形式化模型，并证明目标代码是构造正确的。该方法保持程序简单，无需修改编译器，并在形式化证明中实现了清晰的关注点分离。

Conclusion: Accompanist为编排编程语言提供了一个弹性的运行时，使得在去中心化环境中实现可靠的saga事务成为可能，通过计算重放机制解决了编排编程对网络可靠性和服务正常运行时间的强依赖问题。

Abstract: In service-oriented architecture, services coordinate in one of two ways: directly, using point-to-point communication, or indirectly, through an intermediary called the orchestrator. Orchestrators tend to be more popular because their local state is a 'single source of truth' for the status of ongoing workflows, which simplifies fault recovery and rollback for distributed transactions that use the 'saga' pattern. But orchestration is not always an option because of hardware constraints and security policies. Without a central orchestrator, resilient saga transactions are hard to implement correctly.
  A natural idea is to use choreographic programming, a paradigm that brings the 'global view' of orchestrators to a decentralised setting. Unfortunately, choreographic programming relies on strong assumptions about network reliability and service uptime that often do not hold. Recent work weakens some of these assumptions with 'failure-aware' language features, but these features make programs more complex. We propose a complementary approach: to co-design the programming interface with a customizable runtime that can replay computation to mask faults. Our approach keeps programs simple, does not require modifying the compiler, and lends itself to a clean separation of concerns in formal proofs.
  We present Accompanist, a resilient runtime for the Choral choreographic programming language. With Accompanist, programmers can implement decentralised saga transactions as choreographic programs and deploy the compiled code to 'sidecars' that run alongside services in a pre-existing codebase. Our key assumptions are that choreographic programs should be deterministic, transactions within a saga should be idempotent, and messages should be written to a durable message queue. Based on these assumptions, we present a formal model and prove that target code is correct-by-construction.

### [3] [A Simple and Efficient Implementation of Strong Call by Need by an Abstract Machine](https://arxiv.org/abs/2603.21949)
*Małgorzata Biernacka,Witold Charatonik,Tomasz Drab*

Main category: cs.PL

TL;DR: RKNL是一个实现强按需调用（strong call-by-need）的抽象机，通过自动转换技术从高阶求值器推导而来，具有双线性开销和简洁的实现。


<details>
  <summary>Details</summary>
Motivation: 现有的强按需调用实现要么不够简单，要么效率不高。强按需调用结合了完全正规化和惰性求值的共享特性，但缺乏同时具备简洁性和高效性的实现方案。

Method: 使用记忆thunk技术实现惰性求值的高阶求值器，通过"函数对应"转换工具自动推导出RKNL抽象机。该机器扩展了弱按需调用的Krivine机器，并证明能双线性模拟正规序策略。

Result: RKNL抽象机实现了强按需调用，具有双线性开销（在beta归约次数和输入项大小上都是线性的），并且保守扩展了弱按需调用的Krivine机器。

Conclusion: 通过自动转换技术成功实现了既简单又高效的强按需调用抽象机RKNL，解决了该领域长期存在的简洁性与效率难以兼得的问题。

Abstract: Strong call-by-need combines full normalization with the sharing discipline of lazy evaluation, yet no prior implementation achieved both simplicity and efficiency. We introduce RKNL, an abstract machine that realizes strong call-by-need with bilinear overhead. The machine has been derived automatically from a higher-order evaluator that uses the technique of memothunks to implement laziness. By employing an off-the-shelf transformation tool implementing the ``functional correspondence'' between higher-order interpreters and abstract machines, we obtained a simple and concise description of the machine. We prove that the resulting machine conservatively extends the lazy version of Krivine machine for the weak call-by-need strategy, and that it simulates the normal-order strategy in a bilinear number of steps, i.e., linear in both the number of beta-reductions and the size of the input term.

</details>


### [4] [Set-Theoretic Types for Erlang: Theory, Implementation, and Evaluation](https://arxiv.org/abs/2603.22032)
*Albert Schimpf,Stefan Wehr,Annette Bieniusa*

Main category: cs.PL

TL;DR: 为Erlang设计了一个集合论类型系统，能够静态检查Erlang代码的类型错误，解决了Erlang动态类型导致运行时错误的问题。


<details>
  <summary>Details</summary>
Motivation: Erlang的动态类型系统会导致运行时错误，即使进程重启后这些错误仍然存在。虽然Erlang提供了类型规范语言，但编译器不强制执行这些类型，使得类型规范仅用于文档目的。需要一种能够静态检查Erlang代码类型的方法。

Method: 提出了一个集合论类型系统来捕获Erlang现有类型语言的核心特性。该形式化类型系统保证了类型安全性，并确保类型检查的可判定性。同时实现了一个类型检查器，支持Erlang类型语言的所有特性和大多数术语级语言构造。

Result: 通过案例研究，包括Erlang标准库模块、外部项目以及类型检查器本身的代码，证明了该类型检查器能够有效验证实际Erlang代码。

Conclusion: 该工作成功地为Erlang开发了一个实用的类型检查系统，能够静态检测类型错误，提高了Erlang代码的可靠性，同时保持了类型检查的可判定性。

Abstract: Erlang's dynamic typing discipline can lead to runtime errors that persist even after process restarts. Some of these runtime errors could be prevented through static type checking. While Erlang provides a type specification language, the compiler does not enforce these types, thereby limiting their role to documentation purposes. Type checking Erlang code is challenging due to language features such as dynamic type tests, subtyping, equi-recursive types, polymorphism, intersection types in signatures, and untagged union types. This work presents a set-theoretic type system for Erlang which captures the core features of Erlang's existing type language. The formal type system guarantees type soundness, and ensures that type checking remains decidable. Additionally, an implementation of a type checker is provided, supporting all features of the Erlang type language and most term-level language constructs. A case study with modules from Erlang's standard library, an external project, and the type checker itself demonstrates its effectiveness in verifying real-world Erlang code.

</details>

### [5] [MINISA: Minimal Instruction Set Architecture for Next-gen Reconfigurable Inference Accelerator](https://arxiv.org/abs/2603.20623)
*Jianming Tong,Devansh Jain,Yujie Li,Charith Mendis,Tushar Krishna*

Main category: cs.AR

TL;DR: MINISA提出了一种最小指令集，用于在虚拟神经元粒度上编程可重构AI加速器，大幅减少控制开销并提升性能。


<details>
  <summary>Details</summary>
Motivation: 现代可重构AI加速器依赖丰富的映射和数据布局灵活性来维持高利用率，但通过细粒度微控制暴露这种灵活性会导致从片外内存获取配置位的控制开销过高。

Method: 首先改进FEATHER+消除运行时数据流/布局协同切换的冗余片上复制，支持动态情况。然后将FEATHER+控制抽象为三个布局设置指令（输入、权重、输出虚拟神经元）和一个映射设置指令，减少控制开销同时保留合法映射和布局空间。

Result: MINISA在各种规模下将几何平均片外指令流量减少35倍到4×10^5倍，消除了消耗96.9%微指令周期的指令获取停顿，为16×256 FEATHER+带来高达31.6倍的端到端加速。

Conclusion: MINISA通过虚拟神经元粒度的最小指令集编程，在保留硬件灵活性的同时大幅降低控制开销，显著提升了可重构AI加速器的性能。

Abstract: Modern reconfigurable AI accelerators rely on rich mapping and data-layout flexibility to sustain high utilization across matrix multiplication, convolution, and emerging applications beyond AI. However, exposing this flexibility through fine-grained micro-control results in prohibitive control overhead of fetching configuration bits from off-chip memory. This paper presents MINISA, a minimal instruction set that programs a reconfigurable accelerator at the granularity of Virtual Neurons (VNs), the coarsest control granularity that retains flexibility of hardware and the finest granularity that avoids unnecessary control costs. First, we introduce FEATHER+, a modest refinement of FEATHER, that eliminates redundant on-chip replication needed for runtime dataflow/layout co-switching and supports dynamic cases where input and weight data are unavailable before execution for offline layout manipulation. MINISA then abstracts control of FEATHER+ into three layout-setting instructions for input, weight, and output VNs and a single mapping instruction for setting dataflow. This reduces the control and instruction footprint while preserving the legal mapping and layout space supported by the FEATHER+. Our results show that MINISA reduces geometric mean off-chip instruction traffic by factors ranging from 35x to (4x10^5)x under various sizes under 50 GEMM workloads spanning AI (GPT-oss), FHE, and ZKP. This eliminates instruction-fetch stalls that consume 96.9% of micro-instruction cycles, yielding up to 31.6x end-to-end speedup for 16x256 FEATHER+. Our code: https://github.com/maeri-project/FEATHER/tree/main/minisa.

</details>

### [6] [AC4A: Access Control for Agents](https://arxiv.org/abs/2603.20933)
*Reshabh K Sharma,Dan Grossman*

Main category: cs.CR

TL;DR: AC4A是一个为LLM智能体设计的细粒度访问控制框架，解决了现有系统"全有或全无"的权限问题，让用户能够限制智能体对API和网页内容的访问权限。


<details>
  <summary>Details</summary>
Motivation: 当前LLM智能体系统采用粗粒度的访问控制方式，要么完全访问API或网页的所有功能，要么完全不能访问。这迫使用户必须信任智能体拥有超出任务实际需要的权限，存在安全风险。

Method: AC4A采用分层资源模型，从Unix文件系统等成熟访问控制框架中汲取灵感。应用将其资源定义为层次结构，并在运行时计算访问资源所需的权限。该框架不规定具体权限策略，而是提供灵活的定义和执行机制。

Result: 通过实际案例研究，证明了AC4A能够有效执行对真实API和网页的权限控制。该框架支持API和浏览器两种类型的智能体，源代码已在GitHub开源。

Conclusion: AC4A为LLM智能体提供了一个实用的细粒度访问控制框架，解决了现有系统的权限过度授权问题，使智能体在保持自主性的同时能够受到适当的权限约束。

Abstract: Large Language Model (LLM) agents combine the chat interaction capabilities of LLMs with the power to interact with external tools and APIs. This enables them to perform complex tasks and act autonomously to achieve user goals. However, current agent systems operate on an all-or-nothing basis: an agent either has full access to an API's capabilities and a web page's content, or it has no access at all. This coarse-grained approach forces users to trust agents with more capabilities than they actually need for a given task.
  In this paper, we introduce AC4A, an access control framework for agents. As agents become more capable and autonomous, users need a way to limit what APIs or portions of web pages these agents can access, eliminating the need to trust them with everything an API or web page allows. Our goal with AC4A is to provide a framework for defining permissions that lets agents access only the resources they are authorized to access. AC4A works across both API-based and browser-based agents. It does not prescribe what permissions should be, but offers a flexible way to define and enforce them, making it practical for real-world systems.
  AC4A works by creating permissions granting access to resources, drawing inspiration from established access control frameworks like the one for the Unix file system. Applications define their resources as hierarchies and provide a way to compute the necessary permissions at runtime needed for successful resource access. We demonstrate the usefulness of AC4A in enforcing permissions over real-world APIs and web pages through case studies. The source code of AC4A is available at https://github.com/reSHARMA/AC4A

</details>

### [7] [Instruction Set and Language for Symbolic Regression](https://arxiv.org/abs/2603.21836)
*Ezequiel Lopez-Rubio,Mario Pascual-Gonzalez*

Main category: cs.CL

TL;DR: IsalSR提出了一种符号回归表示框架，通过两层字母表编码表达式DAG为字符串，并计算修剪后的规范字符串来消除结构冗余


<details>
  <summary>Details</summary>
Motivation: 符号回归中存在结构冗余问题：同一个表达式DAG可以有多种不同的节点编号方案，每种都占据搜索空间中的不同点，消耗适应度评估却不增加多样性

Method: 提出IsalSR表示框架，将表达式DAG编码为紧凑两层字母表上的字符串，计算修剪后的规范字符串作为完全标记DAG同构不变量，将所有等价表示折叠为单一规范形式

Result: 该方法能够消除符号回归中的结构冗余，将同一表达式的多个等价表示统一为单一规范形式

Conclusion: IsalSR通过规范表示解决了符号回归中的结构冗余问题，提高了搜索效率

Abstract: A fundamental but largely unaddressed obstacle in Symbolic regression (SR) is structural redundancy: every expression DAG with admits many distinct node-numbering schemes that all encode the same expression, each occupying a separate point in the search space and consuming fitness evaluations without adding diversity. We present IsalSR (Instruction Set and Language for Symbolic Regression), a representation framework that encodes expression DAGs as strings over a compact two-tier alphabet and computes a pruned canonical string -- a complete labeled-DAG isomorphism invariant -- that collapses all the equivalent representations into a single canonical form.

</details>
