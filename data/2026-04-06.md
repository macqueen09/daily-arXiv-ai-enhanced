### [2] [Making Written Theorems Explorable by Grounding Them in Formal Representations](https://arxiv.org/abs/2604.02598)
*Hita Kambhamettu,Will Crichton,Sean Welleck,Harrison Goldstein,Andrew Head*

Main category: cs.HC

TL;DR: 论文提出"可探索定理"系统，将数学定理和证明通过LLM转换为Lean形式化语言，实现交互式证明理解，用户可逐步执行、测试自定义示例并追踪逻辑依赖关系。


<details>
  <summary>Details</summary>
Motivation: LLM生成的解释虽然能提高技术内容可访问性，但存在交互性限制。由于LLM输出是静态文本，无法执行或逐步调试。作者认为，将解释基于形式化表示可以实现超越静态文本的交互功能。

Method: 开发"可探索定理"系统：1) 使用LLM将定理及其书面证明转换为Lean（机器检查证明的编程语言）；2) 将书面证明与Lean代码链接；3) 用户可在步骤级粒度逐步执行证明，测试自定义示例或反例，追踪逻辑依赖关系；4) 每个工作步骤通过执行Lean证明并提取中间状态生成。

Result: 用户研究（n=16）显示：在证明阅读任务中，使用可探索功能的参与者对理解问题给出更好、更正确、更详细的答案，表现出对底层数学更强的整体理解。

Conclusion: 将LLM生成解释与形式化表示相结合，通过可探索定理系统显著增强了数学证明的理解和交互性，为技术内容理解提供了超越静态文本的新方法。

Abstract: LLM-generated explanations can make technical content more accessible, but there is a ceiling on what they can support interactively. Because LLM outputs are static text, they cannot be executed or stepped through. We argue that grounding explanations in a formalized representation enables interactive affordances beyond what static text supports. We instantiate this idea for mathematical proof comprehension with explorable theorems, a system that uses LLMs to translate a theorem and its written proof into Lean, a programming language for machine-checked proofs, and links the written proof with the Lean code. Readers can work through the proof at a step-level granularity, test custom examples or counterexamples, and trace the logical dependencies bridging each step. Each worked-out step is produced by executing the Lean proof on that example and extracting its intermediate state. A user study ($n = 16$) shows potential advantages of this approach: in a proof-reading task, participants who had access to the provided explorability features gave better, more correct, and more detailed answers to comprehension questions, demonstrating a stronger overall understanding of the underlying mathematics.

</details>

### [3] [KAIJU: An Executive Kernel for Intent-Gated Execution of LLM Agents](https://arxiv.org/abs/2604.02375)
*Cormac Guerin,Frank Guerin*

Main category: cs.SE

TL;DR: KAIJU提出了一种系统级抽象，将LLM智能体的工作流执行与推理层解耦，通过意图门控执行和安全内核解决传统ReAct智能体的串行延迟、上下文增长和安全性问题。


<details>
  <summary>Details</summary>
Motivation: 传统基于ReAct的LLM工具调用智能体存在三个主要问题：串行延迟导致效率低下、二次上下文增长增加计算成本、易受提示注入和幻觉攻击。现有方法虽然尝试分离规划与执行，但模型仍与执行机制紧密耦合。

Method: 提出系统级抽象，包含两个核心概念：1) 意图门控执行(IGX)，基于范围、意图、影响和外部批准四个变量进行安全授权；2) 执行内核，负责调度、工具分发、依赖解析、故障处理和安全性。KAIJU采用乐观并行调度和依赖感知参数注入，支持三种自适应执行模式。

Result: 与ReAct基线相比，KAIJU在简单查询上因规划开销有延迟惩罚，在中等复杂度任务上性能相当，在需要并行数据收集的计算查询上具有结构优势。更重要的是，这种分离提供了ReAct仅通过提示无法实现的行为保证。

Conclusion: KAIJU通过解耦规划和执行，不仅提升了复杂任务的执行效率，更重要的是建立了可靠的安全保障机制，为LLM智能体的系统级设计提供了新范式。

Abstract: Tool-calling autonomous agents based on large language models using ReAct exhibit three limitations: serial latency, quadratic context growth, and vulnerability to prompt injection and hallucination. Recent work moves towards separating planning from execution but in each case the model remains coupled to the execution mechanics. We introduce a system-level abstraction for LLM agents which decouples the execution of agent workflows from the LLM reasoning layer. We define two first-class abstractions: (1) Intent-Gated Execution (IGX), a security paradigm that enforces intent at execution, and (2) an Executive Kernel that manages scheduling, tool dispatch, dependency resolution, failures and security. In KAIJU, the LLM plans upfront, optimistically scheduling tools in parallel with dependency-aware parameter injection. Tools are authorised via IGX based on four independent variables: scope, intent, impact, and clearance (external approval). KAIJU supports three adaptive execution modes (Reflect, nReflect, and Orchestrator), providing progressively finer-grained execution control apt for complex investigation and deep analysis or research. Empirical evaluation against a ReAct baseline shows that KAIJU has a latency penalty on simple queries due to planning overhead, convergence at moderate complexity, and a structural advantage on computational queries requiring parallel data gathering. Beyond latency, the separation enforces behavioural guarantees that ReAct cannot match through prompting alone. Code available at https://github.com/compdeep/kaiju

</details>


### [4] [A Synthesis Method of Safe Rust Code Based on Pushdown Colored Petri Nets](https://arxiv.org/abs/2604.02399)
*Kaiwen Zhang,Guanjun Liu*

Main category: cs.SE

TL;DR: 提出基于Pushdown Colored Petri Net的Rust代码合成方法，通过建模所有权、借用和生命周期约束，从API签名自动生成正确且安全的Rust代码。


<details>
  <summary>Details</summary>
Motivation: 安全Rust通过严格的编译时约束保证内存安全，但自动合成正确且安全的Rust代码具有挑战性，因为生成的代码必须同时满足所有权、借用、生命周期约束以及类型和接口要求。

Method: 提出Pushdown Colored Petri Net模型，使用令牌颜色编码动态资源状态和生命周期区域，通过下推栈跟踪生命周期参数的进入和退出，仅当类型匹配、接口义务满足且所需资源状态可用时才启用转换。

Result: 基于双模拟理论证明PCPN的启用和触发规则与这三种约束的编译时检查一致，开发了基于PCPN的自动合成工具，实验结果显示所有合成代码都是正确的。

Conclusion: 提出的PCPN方法能够有效建模Rust的编译时约束，实现从API签名自动合成正确且安全的Rust代码，为解决Rust代码合成挑战提供了有效方案。

Abstract: Safe Rust guarantees memory safety through strict compile-time constraints: ownership can be transferred, borrowing can temporarily guarantee either shared read-only or exclusive write access, and ownership and borrowing are scoped by lifetime. Automatically synthesizing correct and safe Rust code is challenging, as the generated code must not only satisfy ownership, borrowing, and lifetime constraints, but also meet type and interface requirements at compile time. This work proposes a synthesis method based on our newly defined Pushdown Colored Petri Net (PCPN) that models these compilation constraints directly from public API signatures to synthesize valid call sequences. Token colors encode dynamic resource states together with a scope level indicating the lifetime region in which a borrow is valid. The pushdown stack tracks the entering or leaving of lifetime parameter via pushing and popping tokens. A transition is enabled only when type matching and interface obligations both hold and the required resource states are available. Based on the bisimulation theory, we prove that the enabling and firing rules of PCPN are consistent with the compile-time check of these three constraints. We develop an automatic synthesis tool based on PCPN and the experimental results show that the synthesized codes are all correct.

</details>


### [5] [TypePro: Boosting LLM-Based Type Inference via Inter-Procedural Slicing](https://arxiv.org/abs/2604.02702)
*Teyu Lin,Minghao Fan,Huaxun Huang,Zhirong Shen,Rongxin Wu*

Main category: cs.SE

TL;DR: TypePro：基于LLM的动态语言类型推断方法，通过过程间代码切片补充上下文信息，提升类型推断准确率


<details>
  <summary>Details</summary>
Motivation: 动态语言（如Python和JavaScript）虽然灵活，但容易产生类型相关错误，现有类型推断方法仅关注变量的直接依赖关系，导致上下文信息不完整，影响推断准确性

Method: 提出TypePro方法：1）利用LLM进行动态语言类型推断；2）通过过程间代码切片补充上下文信息；3）基于切片中数据类型隐含的结构信息提出候选复杂类型，弥补LLM领域知识不足

Result: 在ManyTypes4Py和ManyTypes4TypeScript数据集上，Top-1精确匹配率分别达到88.9%和86.6%，相比次优方法分别提升7.1和10.3个百分点

Conclusion: TypePro通过补充上下文信息和提出候选复杂类型，有效解决了动态语言类型推断中上下文信息不完整的问题，显著提升了类型推断的准确性和鲁棒性

Abstract: Dynamic languages (such as Python and JavaScript) offer flexibility and simplified type handling for programming, but this can also lead to an increase in type-related errors and additional overhead for compile-time type inference. As a result, type inference for dynamic languages has become a popular research area. Existing approaches typically achieve type inference through static analysis, machine learning, or large language models (LLMs). However, current work only focuses on the direct dependencies of variables related to type inference as the context, resulting in incomplete contextual information and thus affecting the accuracy of type inference. To address this issue, this paper proposes a method called TypePro, which leverages LLMs for type inference in dynamic languages. TypePro supplements contextual information by conducting inter-procedural code slicing. Then, TypePro proposes a set of candidate complex types based on the structural information of data types implied in the slices, thereby addressing the lack of domain knowledge of LLMs. We conducted experiments on the ManyTypes4Py and ManyTypes4TypeScript datasets, achieving Top-1 exact match (EM) rates of 88.9% and 86.6%, respectively. Notably, TypePro improves the Top-1 Exact Match by 7.1 and 10.3 percentage points over the second-best approach, showing the effectiveness and robustness of TypePro.

</details>
