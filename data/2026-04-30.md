### [1] [Finite Functional Programming](https://arxiv.org/abs/2604.26161)
*Michael Arntzenius,Max Willsey*

Main category: cs.PL

TL;DR: 本文提出"有限函数式编程"范式，通过将谓词视为带支持集（非零输出输入集）的函数，统一了函数式与逻辑式编程，并利用分级效应和相关性类型检查有限支持性。


<details>
  <summary>Details</summary>
Motivation: 统一函数式编程和逻辑式编程两种范式，处理Datalog、聚合运算和加权逻辑编程等场景，使谓词可作为数据（输入输出表）与高阶函数（代码）组合。

Method: 将谓词视为带有支持集（非零输出对应的输入集合）的函数；从布尔函数推广到其他带点集（pointed sets）；使用分级效应（graded effects）检查变量接地（grounding），使用相关性类型（relevance types）建模带点集。

Result: 提出了"有限函数式编程"（finite functional programming）的概念，给出了一个简单的类型系统来确保函数的有限支持性。

Conclusion: 通过有限支持函数统一函数式与逻辑式编程，为Datalog、聚合操作和加权逻辑编程提供了统一的理论基础。

Abstract: We unify functional and logic programming by treating predicatesas functions equipped with their support: the set of inputs whose output is nonzero. Datalog, for instance, is a language of finitely supported boolean functions. Finite support allows representing functions as input-output tables. Generalizing from boolean functions to other pointed sets neatly handles aggregation and weighted logic programming. We refer to the combination of finitely supported functions, represented as data, with higher order functions, represented as code, as finite functional programming. We give a simple type system to check finite support, using graded effects to check variable grounding and relevance types to model pointed sets.

</details>

### [2] [Large Language Models for Multilingual Code Intelligence: A Survey](https://arxiv.org/abs/2604.25960)
*Chao Jiang,Dugang Liu,Cheng Wen,Zhiwu Xu,Hua Zheng,Muhammad Sadiq,Jawwad Ahmed Shamsi,Shengchao Qin,Zhong Ming*

Main category: cs.SE

TL;DR: 本文综述了多语言代码生成和代码翻译任务，指出现有研究偏向Python等高级语言，对Rust、OCaml等语言支持较弱，并讨论了可信跨语言泛化的挑战与机遇。


<details>
  <summary>Details</summary>
Motivation: 当前AI辅助软件工程研究偏向Python等高级语言，对Rust、OCaml等语言支持较弱，而真实系统本质上是多语言的，因此需要鲁棒的多语言代码智能能力。

Method: 综述多语言代码生成（从共享自然语言需求生成代码）和多语言代码翻译（保持语义的跨语言转换）两项关键任务，回顾代表性方法、基准数据集和评估指标。

Result: 指出现有方法在低资源语言上表现较弱，多语言代码生成和翻译仍面临挑战，跨语言泛化的可信度有待提升。

Conclusion: 强调实现可信的跨语言泛化是多语言代码智能的关键挑战，为未来研究指明了方向。

Abstract: Large language models have transformed AI-assisted software engineering, but current research remains biased toward high-resource languages such as Python, with weaker performance in languages like Rust and OCaml. Since real-world systems are inherently polyglot, robust multilingual code intelligence is crucial. This survey focuses on two key tasks: multilingual code generation from shared natural-language requirements, and multilingual code translation that preserves semantics across languages. It reviews representative methods, benchmarks, and evaluation metrics, and highlights challenges and opportunities for trustworthy cross-language generalization.

</details>


### [3] [Comparing Smart Contract Paradigms: A Preliminary Study of Security and Developer Experience](https://arxiv.org/abs/2604.26727)
*Matteo Vaccargiu,Andrea Pinna,Maria Ilaria Lunesu,Giuseppe Destefanis*

Main category: cs.SE

TL;DR: 通过对12组功能等价的Solidity和Move智能合约进行混合方法研究，发现资源导向型语言Move通过类型系统将安全开销从运行时转移到编译期，减少了60%的安全检查密度，但代码体积增加47%，开发者学习曲线中等但安全信心更高。


<details>
  <summary>Details</summary>
Motivation: 智能合约漏洞已造成数十亿美元损失，亟需验证编程语言范式（如资源导向型语言Move）能否从语言层面降低安全负担，减少开发者手动实现安全检查的认知开销。

Method: 采用混合方法研究：对同一团队开发的12组功能等价合约（Solidity vs Move）进行定量分析（安全检查密度、代码体积、圈复杂度），并对11名双语言经验开发者进行问卷调查。

Result: Move的安全检查密度显著降低60%（6.7% vs 16.8%，p=0.002, d=-1.75），但代码体积增加47%（p=0.002, d=1.90）；圈复杂度无显著差异。开发者安全信心中位数6/7，55%倾向为安全关键应用选择Move。

Conclusion: 资源导向范式可将安全保证从运行时验证转移到编译时保证，但采用需投入学习成本和工具链建设，初步证据支持语言范式对智能合约开发安全性的积极影响。

Abstract: Smart contract vulnerabilities have caused billions in financial losses, raising questions about whether programming language paradigms can reduce security overhead. While imperative languages like Solidity require developers to manually implement security checks, resource-oriented languages like Move encode safety guarantees in type systems. We present a preliminary mixed-methods study analyzing 12 functionally-equivalent contract pairs implemented in both Solidity and Move by the same development team, complemented by a survey of 11 developers experienced in both languages. Quantitative analysis reveals that Move reduces explicit security overhead by 60\% (security check density: 6.7% vs. 16.8%, p=0.002, Cohen's d=-1.75) at the cost of 47% larger code size (p=0.002, d=1.90), while maintaining identical cyclomatic complexity. Developer surveys show moderate learning difficulty but higher safety confidence in Move (Median=6/7, 10 of 11 above neutral), with 55% preferring Move for security-critical applications despite ecosystem maturity gaps. These preliminary findings suggest resource-oriented paradigms shift security from runtime validation to compile-time guarantees, though adoption requires investment in learning and tooling. The controlled comparison provides initial evidence for paradigm effects on smart contract development, informing language selection decisions and identifying opportunities for improved developer resources.

</details>
