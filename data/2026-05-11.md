### [1] [Bounding Fixed Points of Non-Monotone Processes: Theory to Practice](https://arxiv.org/abs/2605.06803)
*Abdullah H. Rasheed,Vijay K. Garg*

Main category: cs.PL

TL;DR: 本文针对非单调算子不动点近似计算中的效率低下和近似粗糙问题，提出了基于抽象解释的近似方法，并通过可控非完备性设计了分区收紧算法，在回答集编程和推测性程序分析中验证了其实用性。


<details>
  <summary>Details</summary>
Motivation: 现代求解器和程序分析器依赖非单调推理（如否定即失败、推测性更新、回溯），但经典单调不动点方法不适用。现有近似不动点理论（AFT）虽能提供近似，但实际中近似结果粗糙且计算开销大，亟需改进以使其在编程语言领域实用化。

Method: (1) 证明抽象解释用于近似算子的合理性，以解决效率问题；(2) 引入可控非完备性，设计分区收紧算法，改进AFT最佳近似的精度，保证在有限高度格上的完备性、任意时刻可用性和终止性；(3) 进一步修改确保多项式时间复杂度。

Result: 在回答集编程中作为收敛加速预处理器，在推测性程序分析中减少回滚的同时保持完备性，通过实现层面细节展示了方法的实用性和有效性。

Conclusion: 本文成功将AFT近似方法实用化，兼顾了效率与精度，为非单调推理在编程语言领域的应用提供了理论上合理、实践中高效的解决方案。

Abstract: Many modern solvers and program analyzers rely on non-monotone reasoning (e.g. negation-as-failure, speculative updates, backtracking) for which classical monotone fixed-point methods do not apply. The general problem of finding the fixed points of these processes is a difficult one. For this reason, there have been theoretical efforts in existing Approximation Fixpoint Theory (AFT) from the domain of logic programming to approximate fixed points of non-monotone operators. Tight approximations of these fixed points are highly useful for accelerating non-monotonic computations by restricting the search space. In practice, however, even the best approximations obtained through AFT can be coarse and computationally expensive. We aim to address both issues to make AFT approximation methods practical for use in programming languages (PL) settings. To mitigate inefficiency, we prove the soundness of an abstract interpretation for approximating operators. To improve upon coarse approximations, we carefully introduce controlled unsoundness to design an effective yet practical algorithm for partitioning and tightening AFT's best approximations. This algorithm is sound, anytime, and guarantees termination on finite-height lattices. We further present a modification that ensures polynomial-time complexity. We instantiate these methods in two settings: (1) answer set programming, where it serves as a convergence-accelerating pre-processor, and (2) speculative program analysis, where it reduces rollback while preserving soundness. In both settings, we focus on implementation-level details to demonstrate the practical applicability of our methods.

### [2] [A New Interaction Concept for Interactive and Autoactive Program Verification](https://arxiv.org/abs/2605.06972)
*Wolfram Pfeifer,Mattias Ulbrich,Daniel Drodt*

Main category: cs.PL

TL;DR: 本文提出了一种新颖的交互式程序验证概念，使用户能够在源代码和规约级别上检查和干预证明状态，并基于KeY验证引擎实现了原型插件，用户研究表明该方法有助于理解证明状态和发现缺陷。


<details>
  <summary>Details</summary>
Motivation: 程序验证本质上是不可判定且困难的任务，现有验证器要么需要用户直接在代码中写注解（自动模式，无法检查证明状态或干预），要么只允许在比用户规约更低层的逻辑编码上进行交互，存在心智鸿沟。

Method: 提出一种新的交互概念，允许用户在源代码和规约级别检查和干预证明状态，最小化不同表示之间的心智差距；并在Java验证引擎KeY上以插件形式实现该原型。

Result: 通过用户研究证明该原型能帮助用户理解证明状态，并发现源代码或规约中的缺陷。

Conclusion: 在源代码和规约层面进行交互式验证是可行的，可以有效降低用户心智负担，提升验证效率和缺陷发现能力。

Abstract: Fully functional program verification is an undecidable$\unicode{x2014}$and, hence, inherently difficult$\unicode{x2014}$task, that is not automatically solvable but typically requires user interaction and guidance. Existing verifiers either work autoactively, requiring the user to write annotations in source code, without the possibility to inspect the proof state or intervene in case of an unsuccessful attempt, or allow interactions on a logical encoding that is on a lower level than the user-provided specifications. We present a novel interaction concept which allows the user to inspect and interact with the proof state on source code and specification level. This minimizes the mental gap between the representations. We provide an implementation of the concept as a plugin for the Java verification engine KeY, and show with a user study that this prototype can be beneficial for users to understand the proof state and find defects in source code or specifications.

### [3] [CktFormalizer: Autoformalization of Natural Language into Circuit Representations](https://arxiv.org/abs/2605.07782)
*Jing Xiong,Qi Han,Chenchen Ding,He Xiao,Zunhai Su,Chaofan Tao,Ngai Wong*

Main category: cs.CL

TL;DR: CktFormalizer是一个基于Lean 4的框架，通过依赖类型将硬件缺陷转化为编译时错误，在保持仿真通过率的同时实现95-100%的后端可实现率，并通过自动化定理证明支持PPA优化（面积减少35%，功耗降低30%）。


<details>
  <summary>Details</summary>
Motivation: LLM生成的Verilog代码虽然能通过语法检查，但常存在位宽不匹配、组合环路和case逻辑不完整等问题，导致在后端综合和布线阶段失败（20%的正确设计在综合布线中丢失）。现有方法缺乏形式化保证来捕获这些缺陷。

Method: 将LLM生成的硬件描述重定向到嵌入在Lean 4中的依赖类型HDL，利用Lean三重角色：类型检查器（编码位宽约束、case覆盖和无环性）、正确性防火墙（编译后设计无结构缺陷）、证明助手（构造机器验证的等价性证明，支持参数化宽度和任意输入序列）。

Result: 在VerilogEval（156问题）、RTLLM（50问题）和ResBench（56问题）上，仿真通过率与直接Verilog生成相当；后端可实现性达95-100%；PPA优化阶段实现最高35%面积缩减和30%功耗降低。

Conclusion: 依赖类型形式化方法可有效解决LLM生成硬件的后端可实现性问题，同时通过定理证明支撑可靠的架构探索与优化，为LLM驱动的硬件设计提供了正确性保证的可行路径。

Abstract: LLMs can generate hardware descriptions from natural language specifications, but the resulting Verilog often contains width mismatches, combinational loops, and incomplete case logic that pass syntax checks yet fail in synthesis or silicon. We present CktFormalizer, a framework that redirects LLM-driven hardware generation through a dependently-typed HDL embedded in Lean 4. Lean serves three roles: (i) type checker:dependent types encode bit-width constraints, case coverage, and acyclicity, turning hardware defects into compile-time errors that guide iterative repair; (ii) correctness firewall:compiled designs are structurally free of defects that cause silent backend failures (the baseline loses 20% of correct designs during synthesis and routing; CktFormalizer preserves all of them); (iii) proof assistant:the agent constructs machine-checked equivalence proofs over arbitrary input sequences and parameterized widths, beyond the reach of bounded SMT-based checking. On VerilogEval (156 problems), RTLLM (50 problems), and ResBench (56 problems), CktFormalizer achieves simulation pass rates competitive with direct Verilog generation while delivering substantially higher backend realizability: 95--100% of compiled designs complete the full synthesis, place-and-route, DRC, and LVS flow. A closed-loop PPA optimization stage yields up to 35% area reduction and 30% power reduction through validated architecture exploration, with automated theorem proof ensuring that each optimized variant remains functionally equivalent to its formal specification.

</details>
