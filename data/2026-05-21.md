### [1] [Multicategorical Semantics for Untyped Effects](https://arxiv.org/abs/2605.21337)
*Ariel Grunfeld,Liron Cohen*

Main category: cs.PL

TL;DR: 本文通过引入Freyd operads和Freyd PROPs，为无类型带副作用的Call-by-Value语言建立了完整的范畴语义，解决了计算替换不可交换带来的障碍，并证明了其可靠性和完备性。


<details>
  <summary>Details</summary>
Motivation: 无类型带副作用的Call-by-Value语言的完全性证明在范畴语义中遇到根本性障碍：由于求值顺序具有语义意义，计算替换（computation substitution）不存在规范的同步替换概念，传统的通过替换构造语法规约的方法失效。

Method: 将单次计算替换（绑定步骤）作为原语，计算替换表示为有限顺序列表的组合。形式化于单对象Freyd-multicategory框架中，提出Freyd operad（分离值的笛卡尔operad和计算的对称Ren-笛卡尔preoperad，由Freyd函子连接），并从中构造对应的Freyd PROP。证明该构造可表示，且在严格单对象设置中是到余域1的左伴随。

Result: 利用诱导的项模型，在弱闭Freyd operad中解释无类型计算lambda演算（包含过程和更高阶函数），证明了可靠性、初始性和完备性，得到了适用于无类型带副作用计算的范畴语义，其广度足以涵盖可实现性导向的模型（如monadic combinatory algebras）。

Conclusion: 该研究为无类型副作用计算提供了量身定制的范畴语义框架，解决了传统方法中计算替换的语义障碍，拓宽了范畴语义的适用范围。

Abstract: Completeness proofs in categorical semantics usually proceed by building a syntactic category whose composition is given by substitution. For untyped effectful Call-by-Value languages, this runs into a basic obstacle: there is no canonical notion of simultaneous substitution of computations, since evaluation order is semantically meaningful. We address this by taking single computation substitutions, that is, binding steps, as primitive, and representing computation substitution by finite sequential lists composed by concatenation. We formalize this idea in a one-object Freyd-multicategorical setting. We introduce Freyd operads, separating a cartesian operad of values from a symmetric Ren-cartesian preoperad of computations, connected by a Freyd functor, and from any Freyd operad we construct a corresponding Freyd PROP of substitutions. We prove that this construction is representable and, in the strict one-object setting, left adjoint to restriction to codomain 1. Using the induced term model, we interpret untyped computational lambda-calculus with procedures and higher-order functions in weakly closed Freyd operads, and prove soundness, initiality, and completeness. This yields a categorical semantics tailored to untyped effectful computation and broad enough to encompass realizability-oriented models such as monadic combinatory algebras.

### [2] [Sutra: Tensor-Op RNNs as a Compilation Target for Vector Symbolic Architectures](https://arxiv.org/abs/2605.20919)
*Emma Leonhart*

Main category: cs.LG

TL;DR: Sutra是一种纯函数式编程语言，其编译器将整个程序（包括控制流、I/O等）beta-归约为一个融合的张量操作图，可直接作为PyTorch神经网络运行。同一个程序既是逻辑程序又是可训练的神经网络，实验证明其在多种嵌入上100%准确解码，且可通过反向传播训练。


<details>
  <summary>Details</summary>
Motivation: 旨在消除逻辑编程与神经网络训练之间的鸿沟，使同一个程序代码既可作为可解释的逻辑程序运行，又能作为可训练的神经网络进行梯度反向传播，实现"可读、可重编译的代码即模型"。

Method: 设计Sutra语言，将程序通过beta-归约编译为单一的融合张量操作图。核心机制包括：旋转绑定/解绑、多项式Kleene三值逻辑（在{-1,0,+1}网格上用Lagrange插值多项式精确表示）、尾递归循环等下译为张量操作。整个程序在冻结的嵌入基板上运行。

Result: (1) 在四种嵌入（含蛋白质语言模型ESM-2）上，宽度k=8时100%准确解码，而标准Hadamard乘积已失效（准确率最低仅2.5%）。(2) 模糊规则分类器从随机初始化（18.7±9.5%，随机基线20%）训练到100.0±0.0%（三颗种子），且可训练出标量增益并写回.su源码，重编译后行为误差约2e-7/logit。

Conclusion: Sutra证明了同一段代码可以同时是逻辑程序和可训练的神经网络，训练后的模型本身是可读、可重编译的源代码，实现了可解释性与可训练性的统一。

Abstract: Sutra is a typed, purely functional programming language whose compiled forward pass is a PyTorch neural network. The compiler beta-reduces the whole program -- primitives, control flow, string I/O -- to one fused tensor-op graph over a frozen embedding substrate. Rotation binding, unbind, bundle, polynomial Kleene three-valued logic, and tail-recursive loops all lower to tensor operations; the Kleene connectives are Lagrange-interpolated polynomials exact on the {-1, 0, +1} truth grid. Validation is one fact tested two ways. (1) The same program runs on four frozen embeddings spanning two modalities -- three text encoders (nomic-embed-text, all-minilm, mxbai-embed-large) and one protein language model (ESM-2) -- and decodes bundles at 100% accuracy through width k=8 on every substrate, where the textbook Hadamard product has already collapsed (2.5% on mxbai-embed-large, 7.5% on all-minilm). (2) PyTorch autograd flows through the actually compiled graph: a fuzzy-rule classifier written in .su trains from random init (18.7 +/- 9.5%; chance = 20%, five classes) to 100.0 +/- 0.0% (three seeds) by backpropagating through the emitted graph, the symbolic source unmodified. A weighted variant additionally trains a scalar cosine gain and writes it back into the .su source as a numeric literal; recompiling reproduces the trained behaviour to ~2e-7 per logit, so the trained model is itself legible, recompilable code. The same artifact is therefore both a logic program and a trainable neural network.

</details>

### [3] [Stdlib or Third-Party? Empirical Performance and Correctness of LLM-Assisted Zero-Dependency Python Libraries](https://arxiv.org/abs/2605.21405)
*Peng Ding,Rick Stevens*

Main category: cs.SE

TL;DR: 本文通过zerodep项目系统研究了仅使用Python标准库重新实现第三方库的可行性与性能，发现大多数场景下stdlib实现性能可达标（2倍以内），且在部分场景因避免第三方库架构开销反而获得5-115倍加速；主要性能瓶颈来自C扩展计算，而非纯Python第三方库本身。


<details>
  <summary>Details</summary>
Motivation: 第三方Python库带来依赖管理、供应链风险和部署摩擦，作者想探索仅用标准库能在多大程度上复现第三方库生态，并衡量正确性和性能代价。

Method: 构建zerodep项目：在LLM辅助下，严格约束（无外部导入、单文件、API兼容、正确性验证）重新实现40+个流行第三方库模块，覆盖12个类别，并进行系统基准测试对比性能。

Result: 大多数(stdlib-only)实现性能在参考库2倍以内；C扩展计算（图像处理、二进制序列化、底层密码学）是主要性能悬崖；部分场景因避免第三方库架构开销获得5-115倍加速。

Conclusion: 标准库在多数场景足够胜任，LLM在严格符号约束下能生成正确高效的代码，但C扩展密集型任务仍需原生库；该研究对无依赖软件工程有重要启示。

Abstract: Third-party Python libraries introduce dependency management overhead, supply chain risk, and deployment friction in constrained environments. A natural question is how much of this ecosystem can be replicated using only Python's standard library -- and at what correctness and performance cost. We address this empirically through zerodep, a growing collection of single-file Python modules, each a stdlib-only reimplementation of a popular third-party library, developed with LLM assistance under strict constraints: no external imports, single file, drop-in API compatibility, and mandatory correctness validation against the reference library. Spanning over 40 modules across 12 categories -- including serialization, networking, cryptography, agent protocols, and text processing -- zerodep provides a controlled testbed for two interrelated questions: (1) Where does the stdlib suffice? and (2) Can LLMs effectively generate correct, performant code under tight symbolic constraints? Systematic benchmarking shows that stdlib-only implementations achieve performance parity (within 2x of the reference) in the majority of cases. The primary performance cliff is C-extension-backed computation (image processing, binary serialization, low-level crypto), not the inherent overhead of pure-Python third-party libraries. Conversely, many widely-used libraries carry architectural overhead that LLM-generated stdlib reimplementations avoid, yielding 5--115x speedups in several categories. We characterize the stdlib capability boundary across complexity tiers and library categories, discuss where LLM-assisted development succeeds and where it requires iterative human correction, and examine implications for dependency-free software engineering at scale. zerodep is open-source at https://github.com/Oaklight/zerodep.

</details>

### [4] [Causal Past Logic for Runtime Verification of Distributed LLM Agent Workflows](https://arxiv.org/abs/2605.20923)
*Benedikt Bollig*

Main category: cs.LO

TL;DR: 本文提出Causal Past Logic (CPL)，一种用于分布式LLM Agent工作流中条件判断和循环守卫的时序逻辑，将运行时监控直接嵌入到协调语言中，取代了传统的执行后日志检查。在ZipperGen框架中实现，并通过向量时钟监视器保证本地计算值与守卫的指称语义一致。


<details>
  <summary>Details</summary>
Motivation: 传统的顺序日志监控无法正确处理异步分布式LLM Agent工作流中的因果关系——在异步执行中，某个决策只能依赖于其生命线上因果可见的事件，而日志中较早出现的事件可能在当时是局部未知的。需要一种新的监控范式。

Method: 将Causal Past Logic (CPL)扩展集成到ZipperGen框架中。CPL是一种小型过去时间时序逻辑，除了标准的过去时态算子（previous, since）外，还允许守卫检查其他生命线上最新因果可见的事件和选定变量。使用向量时钟监视器配合最新值视图，在运行时由所属生命线在线评估。

Result: 证明了局部计算的监视器值与守卫在当前事件下的指称语义一致，使得运行时验证成为协调语言本身的一部分。

Conclusion: 该方法超越了事后日志检查的范式，将分布式工作流的因果感知监控嵌入到执行过程中，实现了更准确、更及时的运行时验证。

Abstract: Distributed LLM agent workflows should not be monitored as if they produced a single sequential log. In an asynchronous execution, a decision can only depend on events that are causally visible to the lifeline that makes it: an event that appears earlier in some log may still be unknown locally. We extend the ZipperGen agent-workflow framework with Causal Past Logic (CPL), a small past-time temporal logic for guards in conditionals and while loops. In addition to standard past-time modalities such as previous and since, a guard can inspect the latest causally visible event of another lifeline and selected variables stored there. The formula is a source-level guard: it is evaluated online by the owner lifeline and can influence control flow at runtime. We give a vector-clock monitor with latest-value views and prove that the locally computed monitor value coincides with the denotational semantics of the guard at the current event. Thus runtime verification becomes part of the coordination language itself, rather than a post-hoc check over an execution log.

</details>
