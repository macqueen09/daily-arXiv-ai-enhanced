### [1] [Agentic Separation Logic Specification Synthesis](https://arxiv.org/abs/2605.27531)
*Tarun Suresh,David Korczynski,Julien Vanegue*

Main category: cs.PL

TL;DR: Spec-Agent 是一个面向大型 C++ 仓库的智能体系统，能够自动合成表达力强且经过充分验证的形式化规约（涵盖命题逻辑、一阶逻辑、分离逻辑等），在百万行级开源代码上实现 85% 的合成成功率，且零误报，性能优于 Claude Code Opus 的同时 token 成本降低 10 倍。


<details>
  <summary>Details</summary>
Motivation: 从程序实现和自然语言中自动推断形式化规约对于重构、转译、优化和验证至关重要，但现有基于 LLM 的方法无法同时满足大规模 C++ 仓库的扩展性、表达系统级代码特性（如动态内存和堆分配数据结构）所需的规约能力，以及系统性地验证规约以排除错误候选这三项要求。

Method: Spec-Agent 采用智能体系统设计，针对四种递进式规约语言（命题逻辑、一阶逻辑、命题分离逻辑、一阶分离逻辑），为每个函数结合静态分析和运行时堆追踪选择合适的目标语言，将现有功能测试泛化为模糊测试工具，并通过反例引导的反馈循环迭代优化 LLM 生成的候选规约。

Result: 在包含数百万行代码的开源 C++ 代码库上，Spec-Agent 为目标函数的 85% 合成了有效规约，在模糊测试和专家验证下未观察到误报，相比 Claude Code Opus 性能提升 4.6 倍，同时 token 成本降低 10 倍。

Conclusion: Spec-Agent 成功解决了大型 C++ 代码库中的规约合成难题，通过多级规约语言设计、静态与动态分析结合以及反例引导迭代优化，在保证高表达力和零误报的同时实现了大规模适用性和极低成本，为该领域提供了有效的解决方案。

Abstract: Specification synthesis, the task of automatically inferring formal specifications from program implementations and natural language, is important for refactoring, transpilation, optimization, and verification, yet remains an open challenge for large C++ repositories. Existing LLM-based approaches fail to simultaneously scale to such repositories, produce specifications expressive enough to capture systems-code features such as dynamic memory and heap-allocated data structures, and systematically validate those specifications to rule out incorrect candidates. We present Spec-Agent, an agentic system for synthesizing expressive, well-validated specifications across large C++ codebases. Spec-Agent targets a ladder of specification languages: propositional logic, first-order logic, propositional separation logic, and first-order separation logic. For each function, Spec-Agent uses static analysis and runtime heap tracing to select the appropriate target specification language, generalizes existing functional tests into fuzz harnesses, and iteratively refines LLM-generated candidates via counterexample-guided feedback. We evaluate Spec-Agent on open source C++ codebases comprising millions of lines of code. Spec-Agent synthesizes valid specifications for 85% of target functions, with no false positives observed under fuzzing and expert validation, outperforming Claude Code Opus 4.6 at 10x lower token cost.

### [2] [FPMoE: A Sparse Mixture-of-Experts Approach to Functional Code Generation](https://arxiv.org/abs/2605.27849)
*Loc Pham,Lang Hong Nguyet Anh,Thanh Le-Cong*

Main category: cs.PL

TL;DR: 针对现有代码生成模型在函数式编程语言（Haskell、OCaml、Scala）上表现不佳的问题，提出基于稀疏MoE架构的FPMoE模型，通过三个语言专用专家+一个共享专家的设计，以3B激活参数达到甚至超越更大模型（如14B/30B）的性能。


<details>
  <summary>Details</summary>
Motivation: 现有LLM代码生成模型主要面向命令式语言，函数式编程语言（FPLs）严重被忽视，即使前沿模型在FPLs上表现也较差。微调方法存在两个问题：单语言微调无法捕获跨语言函数式抽象；多语言合并微调则引入跨语言干扰。

Method: 提出FPMoE，基于稀疏MoE架构：包含三个语言专用路由专家（分别负责Haskell、OCaml、Scala）和一个共享专家（捕获跨语言函数式模式，如单子推理、类型导向编程）。专用专家消除干扰，共享专家保留跨语言抽象。

Result: 在FPEval评估集上，FPMoE显著优于微调基线模型；仅3B激活参数，性能即可匹配甚至超越DeepSeek-Coder-6.7B、Qwen2.5-Coder-14B-Instruct、Qwen3-Coder-30B-A3B等大模型。

Conclusion: 稀疏MoE架构有效解决了函数式编程语言代码生成中专用微调与多语言融合之间的矛盾，在极低激活参数下实现了高效的跨语言函数式代码生成能力。

Abstract: Despite rapid progress in LLM-based code generation, existing models are predominantly trained on imperative languages, leaving functional programming languages (FPLs) such as Haskell, OCaml, and Scala chronically underexplored, with even frontier models performing substantially worse on FPLs. Fine-tuning is a natural remedy, but our experiments show that per-language fine-tuning fails to capture shared functional abstractions, while merged multi-language fine-tuning introduces cross-language interference. To address this, we introduce FPMoE, a lightweight, open-source code generation model built on a sparse Mixture-of-Experts (MoE) architecture with three language-specific routed experts (one each for Haskell, OCaml, and Scala) and a shared expert that captures cross-language functional patterns such as monadic reasoning and type-directed programming. This design resolves both failure modes simultaneously: dedicated experts eliminate interference, while the shared expert preserves abstractions that per-language models miss. On FPEval, FPMoE substantially outperforms fine-tuned baselines and, with only 3B active parameters, matches the performance of much larger models including DeepSeek-Coder-6.7B, Qwen2.5-Coder-14B-Instruct, and Qwen3-Coder-30B-A3B.

</details>


### [3] [Skill-as-Pseudocode: Refactoring Skill Libraries to Pseudocode for LLM Agents](https://arxiv.org/abs/2605.27955)
*Xinze Li,Yuhang Zang,Yixin Cao,Aixin Sun*

Main category: cs.PL

TL;DR: 提出Skill-as-Pseudocode (SaP)方法，将Markdown格式的技能库自动转换为带类型的伪代码，通过确定性验证器确保质量，在ALFWorld基准上显著优于基线方法，同时减少token消耗和LLM调用次数。


<details>
  <summary>Details</summary>
Motivation: 现有的Markdown技能库以自由格式的散文形式存在，导致LLM agent每次检索时都需要重新推导输入模式和调用语法，常陷入"困惑→重新检索→仍然困惑"的低效循环。

Method: SaP方法：对相似的过程性文本段落进行聚类，提取带类型的契约（typed contract），通过四重确定性验证器（覆盖性、绑定性、替换性、风险性）过滤，将通过的契约内联到重写的技能骨架中，同时保留具体的操作模板，为agent提供"技能功能"的类型签名和"如何调用"的具体模板两种互补信号。

Result: 在ALFWorld 134个游戏（未见分割）上，使用gpt-4o-mini，三组随机种子下，SaP在402场配对游戏中获胜82场，Graph-of-Skills (GoS)基线获胜47场（McNemar检验p=8.2e-5）；每个游戏减少22.8±6.4%的输入token和14.5±4.1%的LLM调用。

Conclusion: SaP通过将模糊的散文式技能库转化为结构化的类型化伪代码，有效解决了agent的"困惑循环"问题，在性能显著提升的同时还降低了计算成本。

Abstract: Markdown skill libraries for LLM agents ship as free-form prose, forcing the agent to re-derive both the input schema and the concrete invocation syntax on every retrieval. We observe that this often produces a "confused -> re-retrieve -> still confused" loop in which the agent issues a partially-correct action, receives uninformative environment feedback, and re-retrieves the same prose. We propose Skill-as-Pseudocode (SaP), an automatic conversion of markdown skill libraries into typed pseudocode with deterministic quality control. For each cluster of similar procedural passages drawn from one or more skills, SaP extracts a typed contract and filters it through a four-check deterministic verifier (coverage, binding, replacement, risk). Promoted contracts are inlined into a rewritten skill skeleton together with restored concrete action templates, giving the agent two complementary signals: a typed signature for what the skill does and a concrete template for how to invoke it. On the 134-game ALFWorld unseen split with gpt-4o-mini, pooled across three seeds, SaP wins 82/402 paired games versus 47/402 for the Graph-of-Skills (GoS) baseline (pooled McNemar p = 8.2e-5), at -22.8 +/- 6.4% input tokens and -14.5 +/- 4.1% LLM calls per game.

</details>


### [4] [E-Path: Equality Saturation for Control-Flow Graphs](https://arxiv.org/abs/2605.28694)
*Guillermo Garcia*

Main category: cs.PL

TL;DR: 本文提出E-Path数据结构，将等饱和度推广到控制流图上，以指令序列为基本等价单位，支持非破坏性的循环与控制流优化。


<details>
  <summary>Details</summary>
Motivation: 现有等饱和度系统擅长表达式级别的重写，但难以直接表示控制流图上的等价关系，通常需要先将程序归一化为结构化形式才能优化。

Method: 提出E-Path数据结构，记录编译器中间表示中指令序列之间的等价关系，以指令序列为基本等价单位，在受限的ANF控制流图上实例化原型，但模型本身设计为IR无关。

Result: E-Path支持非破坏性地优化循环及其他控制流结构，同时保留多个等价的程序变体，使经典CFG优化可表达为重写驱动的变换。

Conclusion: E-Path将等饱和度思想扩展到控制流图层面，为编译器优化中处理控制流等价性提供了一种通用且非破坏性的新范式。

Abstract: Modern equality saturation systems excel at expression-level rewrites by exploring large spaces of equivalent programs without suffering from the phase-ordering problem. How- ever, these systems struggle to represent equivalence directly over arbitrary control-flow graphs, often requiring normal- ization into structured or tree-like forms before optimization can occur. We present the E-Path data structure, a prototype frame- work for equality-saturation-style optimization over control- flow graphs. Instead of representing congruence between individual expressions, the E-Path records equivalence be- tween instruction sequences embedded within a compiler intermediate representation. In this prototype, E-Path is in- stantiated over a restricted ANF-based control-flow graph used in a compiler backend, but the model itself is intended to be IR-agnostic. By treating instruction sequences as the fundamental unit of congruence, the E-Path enables non-destructive optimiza- tion of loops and other control-flow structures while preserv- ing multiple equivalent program variants simultaneously. This allows classical CFG optimizations to be expressed as rewrite-driven transformations without destructive mutation of the underlying graph.

### [5] [LACUNA: Safe Agents as Recursive Program Holes](https://arxiv.org/abs/2605.28617)
*Yaoyu Zhao,Yichen Xu,Oliver Bračevac,Cao Nguyen Pham,Frank Zhengqing Wu,Martin Odersky*

Main category: cs.AI

TL;DR: LACUNA提出了一个新的智能体编程模型，通过让模型编写的代码在运行时进行类型检查，弥合了智能体运行时与模型生成代码之间的鸿沟，同时保证安全性。


<details>
  <summary>Details</summary>
Motivation: 现有LLM智能体中，运行时（runtime）与模型编写的代码之间存在割裂——运行时控制循环、上下文和流程，而模型对其几乎没有话语权。这限制了智能体的表达能力，同时也放大了安全风险。

Method: 每个智能体动作是一个类型化调用agent[T](task)，由LLM在执行时填充代码，代码在运行前会进行类型检查。动作要么整体接受要么整体拒绝，拒绝时环境不受影响，编译器诊断驱动重试。该机制还约束动作可使用的工具、数据及其流向。

Result: 在BrowseComp-Plus上，8.6%的生成在执行前被拒绝，平均每次查询0.7次重试，准确率达到27.1%。在τ²-bench上，LACUNA在四个领域的392个任务中解决了76.0%，与基线智能体持平。

Conclusion: LACUNA通过类型检查机制有效地弥合了运行时与模型代码之间的鸿沟，在不牺牲性能的前提下提升了智能体的安全性和可表达性。该模型能够自然地表达ReAct循环、子智能体、技能、并行分解和多模型规划等复杂控制流。

Abstract: LLM agents increasingly act by writing code, yet a split persists between the runtime that drives the agent and the code the model writes. The runtime owns the loop, context, and control flow, and the model has little say over any of them. Letting model-written code shape the runtime itself would make agents more expressive, but it would also sharpen safety problems. A model can be diverted by a prompt injection, call the wrong tool, or fail partway and leave an inconsistent state, and each such failure reaches further when the code shapes the runtime than when it expresses a single action. We present LACUNA, a programming model for agents that closes this split while preserving safety. Each agent action is a typed call $\texttt{agent[T](task)}$ that the LLM fills with code when execution reaches it, and the code is type-checked against the surrounding program before it runs. Because each action is accepted or rejected as a whole, a rejected one leaves the environment untouched, and its compiler diagnostics drive a retry. The same check also bounds which tools and data an action may use and how they flow. Our primitive expresses ReAct loops, sub-agents, skills, parallel decomposition, and multi-model planning as ordinary control flow. We evaluate LACUNA on a collection of test cases, BrowseComp-Plus, and $τ^2$-bench. On BrowseComp-Plus, $8.6\%$ of generations are rejected before execution, with 0.7 retries per query on average, and the agent reaches $27.1\%$ accuracy. On $τ^2$-bench, LACUNA solves $76.0\%$ of $392$ tasks across four domains with a capable model, on par with the baseline agent.

</details>
