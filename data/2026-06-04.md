### [1] [Climbing Up the Semantic Tower -- at Runtime](https://arxiv.org/abs/2606.04034)
*François-René Rideau*

Main category: cs.PL

TL;DR: 本文提出"一等实现"（first-class implementations）这一反射协议，允许在运行时动态切换计算的不同语义层级实现。该协议通过提取形式化规范的计算内容得到，弥合了语义学与运行时反射这两个此前互斥的研究领域。


<details>
  <summary>Details</summary>
Motivation: 现有系统在运行时只能沿着语义塔向下层移动（即更具体的实现层），无法反向上升到更高抽象层级。作者希望解决这一不对称性，让程序能动态观察和切换不同抽象层级的实现。

Method: 从形式化规范及其性质中提取计算内容，设计了一套基于"安全点"（safe points）的反射协议。安全点允许在运行时以高于当前执行层级的抽象级别观察计算，从而实现在语义塔中向上攀升。

Result: 成功实现了"一等实现"反射协议，能够在运行时动态切换正在运行的计算的底层实现，且支持向上遍历语义塔。

Conclusion: 通过形式化方法将语义学与运行时反射相结合，证明了理论与实践可以相互赋能，为更灵活的软件系统设计提供了新思路。

Abstract: Software exists at multiple levels of abstraction, where each more concrete level is an implementation of the more abstract level above, in a semantic tower of compilers and/or interpreters. First-class implementations are a reflection protocol to navigate this tower *at runtime*: they enable changing the underlying implementation of a computation *while it is running*. Key is a generalized notion of *safe points* that enable observing a computation at a higher-level than that at which it runs, and therefore to climb up the semantic tower, when at runtime most existing systems only ever allow but to go further down. The protocol was obtained by extracting the computational content of a formal specification for implementations and some of their properties. This approach reconciles two heretofore mutually exclusive fields: Semantics and Runtime Reflection.

</details>


### [2] [Negative and Fractional Types in the Fidelity Framework](https://arxiv.org/abs/2606.04352)
*Houston Haynes*

Main category: cs.PL

TL;DR: 本文在Native Type Universe（NTU）框架基础上，提出将负类型和分数类型作为原生第一类构造。借鉴James/Sabry和Chen/Sabry的对偶性理论，结合Kennedy的维度类型（阿贝尔群代数模式），在保持可判定性和主类型的前提下，将有理数维度指数引入类型系统。该方案可应用于贝叶斯推理（分数类型表达条件约束）、量子计算（负类型提供类型级伴随）和绝热计算（哈密顿量变形为可逆约束传播），为现有软件生态未直接解决的问题空间提供支撑。


<details>
  <summary>Details</summary>
Motivation: 现有通用计算框架缺乏原生支持负类型和分数类型的底层结构，导致这些对偶性理论无法作为第一类构造直接使用，限制了在贝叶斯推理、量子计算和绝热计算等领域的表达能力。

Method: 基于NTU框架，利用James/Sabry建立的对偶性和Chen/Sabry的紧凑闭范畴解释，结合Kennedy维度类型的阿贝尔群代数模式，将负类型和分数类型作为原生类型构造引入，并设计了支持有理数维度指数的Clef语言语法。

Result: 提出了一种在保持可判定性和主类型的前提下，将负类型和分数类型作为原生构造的类型系统方案，可在NTU框架中实现新的同构关系和更简洁的降级策略解析形式。

Conclusion: NTU框架及其支撑结构天然适合处理当前软件生态未直接解决的问题空间，同时保持良好的开发工效和成熟工具链支持，与框架追求的操作保证一致。

Abstract: Our Native Type Universe (NTU) has been detailed through five previous papers establishing the substrate our framework's compilation pipeline targets across multiple hardware platforms. We have found in the course of that work a deeper reach this foundation makes available: negative and fractional types as native first-class constructs. James and Sabry established these dualities in 2012; Chen and Sabry later developed their categorical interpretation in compact closed categories. These dualities have practical benefit for compute modalities in our Fidelity Framework where extant general purpose compute framings lack the substrate to host them as native constructs. We see practicality with these type forms in preserving decidability and principal types through the abelian-group algebraic pattern Kennedy's dimensional types establish. The resulting isomorphisms would admit new, concise forms of resolution within our novel lowering strategy, and we sketch a notional Clef language syntax that would admit rational dimensional exponents into our algebra. We trace the implications across several problem spaces these type forms would open to our compilation and verification disciplines: Bayesian inference where fractional types would express conditioning obligations, quantum computation (and simulations) where negative types would provide the type-level adjoint, and finally adiabatic computation where the combined discipline would express Hamiltonian deformation as a reversible constraint-propagation process. The inherent structure of our NTU together with the supporting framework appears well-suited to problem spaces that current software ecosystems do not directly address, while keeping approachable development ergonomics and mature tooling aligned with operational guarantees the framework aspires to provide.

</details>

### [3] [Token Budgets: An Empirical Catalog of 63 LLM-Agent Budget-Overrun Incidents, with an Affine-Typed Rust Mitigation as a Case Study](https://arxiv.org/abs/2606.04056)
*Sajjad Khan*

Main category: cs.SE

TL;DR: 论文通过分析21个框架的63个生产事故，建立了8类预算超支故障分类体系，并开发了一个1180行Rust库，利用仿射类型系统在编译期防止预算克隆、双重消费和委托后使用等问题，实验表明该方法在多代理场景下零超支，而传统方案30/30超支。


<details>
  <summary>Details</summary>
Motivation: LLM智能代理在生产中频繁出现预算超支问题（如单个重试循环可花费数千美元），而现有类型系统缺乏对成本承载值的别名、双重消费、委托后使用等完整性的编译期保障，依赖临时包装器在运行时防范，操作者极易疏忽。

Method: 1. 收集21个编排框架(2023-2026)的63个生产事故，构建8类故障分类体系(Cohen's kappa=0.837)；2. 开发token-budgets：一个1180行Rust crate(无unsafe)，将仿射所有权机制引入预算管理，使克隆、双重消费、委托后使用成为编译错误而非运行时风险；3. 在多运行时(5种)、多提供商(3个)下进行160次温度分层实时API测试。

Result: 故障分类：63起事故+47个结构性条目，11起与委托扇出竞争相关；token-budgets：单代理任务4行Python计数器即可0/30超支，多代理委托场景下Rust借用检查器编译期拒绝11起事故模式（asyncio方案30/30超支，三种严格替代方案0/30超支）；160次测试零额度违规、零假拒绝；静态过度预留4-6倍(自适应2.11倍)。

Conclusion: 仿射类型系统能有效在编译期防止LLM代理预算超支问题，特别是多代理委托场景下的竞争条件，但运行时二进制级别的额度健全性仍待解决。

Abstract: LLM-agent budget overruns are a documented production failure class: a single retry loop can spend thousands of dollars before an operator notices, and the in-process integrity properties that would prevent it (no aliasing, no double-spend, no use-after-delegation of a cost-bearing value) are enforced, if at all, by ad-hoc wrappers rather than by the type system. Our central contribution is empirical: a catalog of 63 confirmed production incidents from 21 orchestration frameworks (2023-2026), each backed by a quoted GitHub issue and, where reported, a dollar loss, organized into an eight-cluster failure taxonomy (inter-rater Cohen's kappa = 0.837, N = 113), plus 47 supplementary structural entries. As one mitigation evaluated against this taxonomy, we build token-budgets, an 1,180-line Rust crate (no unsafe) that operationalizes affine ownership so that cloning, double-spending, or using a budget after delegating it are compile errors rather than runtime hazards an operator must remember to avoid. The dollar cap is runtime arithmetic under an estimator assumption; the affine layer makes that arithmetic non-bypassable. On single-agent workloads a 4-line Python counter matches the crate at 0/30 overshoot, so the distinguishing value is non-bypassability under operator error in multi-agent delegation: the delegation-fanout race documented in 11 incidents is rejected by the borrow checker at compile time, while the same pattern under asyncio overshoots 30/30 and three disciplined alternatives overshoot 0/30. Across five runtimes, three providers, and a temperature-stratified live-API test (N = 160), the approach reports zero cap violations and zero false refusals, at operational parity with concurrent work. Static over-reservation is 4-6x (2.11x adaptive). Binary-level cap-soundness on the running binary is left open.

</details>

### [4] [Formal verification of the S-two AIR](https://arxiv.org/abs/2606.04311)
*Jeremy Avigad,Anat Ganor,Lior Goldberg,David Levit,Ohad Nir,Yoav Seginer,Alon Titelman*

Main category: cs.CR

TL;DR: 本文使用Lean 4证明助手，形式化验证了StarkWare的S-two证明系统中AIR编码的可靠性，即AIR的可满足性蕴含程序运行完成的计算声明。


<details>
  <summary>Details</summary>
Motivation: 确保区块链上Cairo虚拟机程序运行的声称可通过AIR编码被正确表达，避免因编码错误导致证明系统虽通过验证但实际计算声明不成立的风险。

Method: 在Lean 4证明助手中，对StarkWare的S-two证明系统所使用的代数中间表示（AIR）进行形式化建模，并证明其可满足性蕴含底层Cairo程序执行完成性的计算声明。

Result: 成功在Lean 4中完成了对AIR编码可靠性的形式化验证，确认了AIR可满足性与计算声明之间的蕴含关系成立。

Conclusion: 该形式化验证为StarkWare的证明系统提供了更高层次的可信度，确保AIR编码在语义上是正确的，从而增强了Cairo虚拟机程序在区块链上运行证明的可靠性。

Abstract: StarkWare's S-two prover provides an efficient means for establishing, on blockchain, that a program written in the Cairo virtual machine language runs to completion. The latter claim is encoded by an algebraic intermediate representation (AIR) that captures the semantics of the Cairo language. The AIR asserts the existence of tables of values from a finite field satisfying certain algebraic constraints. A cryptographic interactive proof system, circle STARK, provides an efficiently-checked certificate that the AIR is satisfied. We describe our verification, using the Lean 4 proof assistant, that the AIR encoding is sound, which is to say, the satisfiability of the AIR implies the computational claim.

</details>

### [7] [Provably Auditable and Safe LLM Agents from Human-Authored Ontologies](https://arxiv.org/abs/2606.04903)
*Aaron Sterling*

Main category: cs.LO

TL;DR: 本文提出了一种名为 Agentic Redux 的 LLM 智能体架构，该架构通过类型化 λ 演算保证了执行语义的正确性，并设计了不可篡改的审计日志。同时提出本体优先的智能体设计方法（Ontology-First Agent Design），并在医疗账单合规与安全漏洞披露两个实际领域进行了验证。


<details>
  <summary>Details</summary>

Method: 1) 构建 Agentic Redux 架构，使用类型化 λ 演算形式化证明语义正确性，采用追加式账本记录所有决策。2) 提出 Ontology-First 方法，先由人类专家用 Basic Formal Ontology（基础形式本体）对领域进行本体化，再让 LLM 推导出智能体与人类在环的角色分工。

Result: 在医疗账单合规与安全漏洞披露两个生产级领域成功实现了可审计、语义正确的智能体执行，提供了可复现的代码仓库。

Abstract: We introduce the LLM agent architecture Agentic Redux, intended for use with nontrivial problem domains that require linear auditability. Using the typed lambda calculus, we prove that, run on appropriate domains, Agentic Redux executions are semantically guaranteed to be correct, with all decisions recorded in an append-only ledger. We present two production-grade appropriate domains, in healthcare billing compliance, and security vulnerability disclosure. Working code for Agentic Redux run on both domains is available in a supporting code repository. We also introduce Ontology-First Agent Design, a methodology for creation of agent frameworks on a problem domain, in which a human expert ontologizes the problem domain with Basic Formal Ontology, and then assigns an LLM to derive roles that agents and humans-in-the-loop can fill, in order to work the problems in the domain.

</details>
