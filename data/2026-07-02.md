### [1] [Certificate-Carrying Transformation of Event-Driven Block Programs](https://arxiv.org/abs/2607.00563)
*Yuan Si,Jialu Zhang*

Main category: cs.PL

TL;DR: 本文提出了一种基于证书携带的源到源重写方法，用于确保块端用户语言（如Scratch）中程序优化的行为保持性。一个不可信优化器提出重写，一个可信检查器通过重新计算每个侧条件来接受或拒绝，且检查器是唯一权威。在Lean中形式化了一个协作帧精化定理，覆盖两种具体重写族，并在300个真实Scratch项目上评估，检查器接受率94.3%，零误接受，成本低于0.1秒/项目。


<details>
  <summary>Details</summary>
Motivation: 现有块端用户语言（如Scratch）的优化工具通过程序分析和测试来保证行为保持，但缺乏经过验证的保证。本文希望提供一种有形式化保证的方法，防止优化器错误导致不合理的接受。

Method: 将优化转化为证书携带的源到源重写：不可信优化器提出重写，可信检查器重新计算每个侧条件，基于协作帧精化定理（在Lean中形式化），观察透镜作为参数，检查器是唯一权威。

Result: 对300个真实Scratch项目评估：检查器接受94.3%（283/300）；认证成本低于0.1秒/项目；跨族对抗测试4278个扰动重写产生零误接受；审计发现8个误接受（现已被拒绝）；消融实验（仅保留分析和测试）失败。

Conclusion: 该方法能为并发、事件驱动的端用户语言提供行为保持保证，检查器重新计算每个所需条件而非信任优化器声明，保持可信基小。

Abstract: Block-based end-user languages such as Scratch run tens of millions of programs. Existing tools establish behavior preservation through program analysis and testing without a checked guarantee. We turn optimization into certificate-carrying source-to-source rewriting. An untrusted optimizer proposes a rewrite; a trusted, fail-closed checker accepts it only after recomputing every side condition that the rewrite's behavior preservation depends on under an explicit observation lens. The checker is the sole authority: given a correct checker and a small, explicitly stated set of model-to-VM assumptions, an optimizer bug cannot mint an unsound acceptance. The observation lens is a parameter, and the central soundness argument is a cooperative-frame refinement theorem: a write overwritten before any thread observes it, within a window in which no thread yields, can be removed. We mechanize this theorem in Lean and show that one parametric statement covers two concrete rewrite families instantiated to variable state and renderer state. We build a checker for six rewrite families and evaluate it on 300 real Scratch projects. The checker accepts a behavior-preserving rewrite on 94.3% of projects (283 of 300); certification costs under one tenth of a second per project; and a cross-family adversarial campaign of 4,278 perturbed rewrites produces zero false accepts. An audit found eight false accepts the per-family test suites missed; each is now rejected. An ablation that strips the semantic side conditions, leaving analysis and testing alone, ships rewrites the virtual machine confirms change behavior; the full checker rejects every one. The result shows how to provide behavior-preservation guarantees for a concurrent, event-driven, end-user language. The checker recomputes every required condition instead of trusting optimizer claims, keeping the trusted base small.

### [2] [Practical Range Refinement Types with Inference](https://arxiv.org/abs/2607.00824)
*Valentin Aebi,Carlo A. Furia*

Main category: cs.PL

TL;DR: Ranger是一种面向整数范围类型的细化类型系统，通过双向类型推断和轻量级流敏感分析，在降低注解开销的同时实现了对数组边界等属性的静态验证，与Java Checker Framework和Liquid Java相比具有优势。


<details>
  <summary>Details</summary>
Motivation: 现有细化类型系统虽然表达能力强大，但注解开销过高，阻碍了实际应用。作者希望设计一个既能保持表达力又能大幅减少用户注解负担的细化类型系统，以推动此类技术在实践中落地。

Method: 提出Ranger系统，专注于整数范围类型。采用双向类型系统（bidirectional type system）结合类型推断算法，自动推断信息以减少用户注解；同时整合轻量级流敏感静态分析技术，无需显式注解即可精确捕获程序行为。

Result: 在Licorne语言上实现Ranger，实验表明：能够简洁表达和验证超出Java/Scala标准类型系统的各种有用属性（如索引正确性、数组越界检查）；与Java Checker Framework和Liquid Java等同类系统相比，Ranger具有更优的性能或更低的注解开销。

Conclusion: Ranger成功地在表达力与易用性之间取得了更好的平衡，证明了通过双向类型推断和流敏感分析可以显著降低细化类型系统的注解负担，为细化类型的实际应用提供了可行方案。

Abstract: Refinement types are a static verification technique that aims at increasing the expressivity of traditional type systems while remaining easy and natural to use. While systems based on refinement types have been developed for several mainstream languages, their practical adoption remains limited by their annotation overhead, which is often a more significant burden than when using the "plain" type annotations of languages like Java or Scala.
  To improve the state of the art, this paper introduces Ranger: a refinement type system designed to keep the annotation overhead small and to seamlessly integrate with imperative-style constructs like variables and loops. As the name suggests, Ranger focuses on integer range types: a particular kind of refinement types that express bounded integer ranges. Such types are widely useful to verify correct index manipulation and in-bounds data accesses, among others. To combine expressiveness and succinctness, Ranger is based on a bidirectional type system, which runs a type inference algorithm to provide the typechecking pass with information useful to reduce the need for user-written auxiliary annotations. Ranger also integrates other forms of lightweight flow-sensitive static analysis techniques that precisely capture the program's behavior without explicit annotations.
  We implemented Ranger on top of the Licorne experimental programming language. Our experiments show that Ranger's implementation can concisely express and verify a variety of useful properties that fall beyond the capabilities of standard static type systems like those of Java and Scala, and that Ranger compares favorably to other extended type systems, such as the Java Checker Framework and Liquid Java, that can also check properties about ranges.

### [3] [AlgoBench: Benchmarking Algorithmic Adaptation in Code Generation](https://arxiv.org/abs/2607.00062)
*Xinyuan Song,Zekun Cai,Liang Zhao*

Main category: cs.SE

TL;DR: 提出ALGOBENCH框架，通过约束变换自动生成新算法问题，评估模型在功能正确性之外的算法适应能力，发现LLM在变体问题上性能显著下降。


<details>
  <summary>Details</summary>
Motivation: 现有编程基准（如HumanEval）存在数据污染问题，模型可能因暴露而非真正算法能力取得高分，需要一种能评估模型算法推理和适应能力的新方法。

Method: ALGOBENCH通过结构化约束移位变换，从已知竞赛题自动生成变体，使原参考算法失效；引入OPTT、OPTS、TRAPRATE、GAPT、CONSENS等复杂度感知指标。

Result: 多个LLM在ALGOBENCH变体上性能显著下降；检索增强会导致旧算法复用；许多正确解未达所需复杂度；错误主要是算法层面而非实现层面。

Conclusion: ALGOBENCH能有效评估模型超越功能正确性的算法适应能力，揭示LLM在算法推理上的局限性。

Abstract: High pass rates on established programming benchmarks such as HumanEval and LiveCodeBench do not always show whether a model can reason about algorithms. Many fixed benchmarks eventually become part of the public training ecosystem through released problem statements, editorials, and generated solutions, allowing later models to improve partly by exposure rather than by stronger algorithmic ability. We introduce ALGOBENCH, a framework that automatically builds novel algorithmic problems from known competitive-programming problems through structured constraint-shifting transformations. Each accepted ALGOBENCH variant is traceable to a source problem, but must make the original reference algorithm fail. Beyond pass@$k$, we introduce complexity-aware metrics -- including OPTT, OPTS, TRAPRATE, GAPT, and CONSENS -- to test whether a solution is not only functionally correct but also asymptotically suitable for the generated problem. Experiments across multiple LLMs and prompting strategies show that performance drops sharply on ALGOBENCH variants, retrieval can increase reuse of the old algorithm, and many correct-looking solutions fail to meet the required complexity. Error analysis shows that failures are mainly algorithmic rather than implementation-level, suggesting that ALGOBENCH evaluates adaptation beyond functional correctness.

### [4] [LLVM-Bench: Benchmarking and Advancing Large Language Models for LLVM Compiler Issue Resolution](https://arxiv.org/abs/2607.00700)
*Zhao Tian,Yingquan Zhao,Chenyao Suo,Meng Wang,Junjie Chen*

Main category: cs.SE

TL;DR: 本文提出了LLVM-Bench（首个大型LLVM编译器问题解决基准，含423个真实任务）和LLVM-Gym（自动化评估平台），并通过实验发现现有LLM方法在编译器问题解决上效果有限，随后提出LLVM-Ens集成方法将解决率提升至21.99%。


<details>
  <summary>Details</summary>
Motivation: LLVM编译器规模大、复杂度高，问题解决耗时费力。虽然LLM在通用问题解决上表现出色，但针对系统级编译器（如LLVM）的有效性尚未被充分探索，缺乏专门的基准和评估平台。

Method: 1) 构建LLVM-Bench：收集423个真实LLVM项目问题。2) 开发LLVM-Gym：自动化复现、打补丁、编译和测试。3) 评估4种LLM、6种检索配置和3种智能体。4) 提出LLVM-Ens：集成不同技术生成的补丁，过滤错误和冗余，选出最优解。

Result: 现有LLM方法在LLVM-Bench上表现有限，主要失败模式为补丁无效和构建失败。不同LLM和智能体间存在强互补性。LLVM-Ens集成方法将解决率提升至21.99%。

Conclusion: 编译器问题解决对LLM仍是重大挑战，但通过多模型集成可以显著提升效果，为后续研究提供了基准和方向。

Abstract: LLVM is a widely used compiler infrastructure whose scale and complexity make issue resolution labor-intensive and challenging. Although large language models (LLMs) have recently achieved remarkable success in issue resolution, their effectiveness on complex system-level LLVM compiler remains largely unexplored. To address this gap, we introduce LLVM-Bench, the first large-scale benchmark for LLVM issue resolution, containing 423 real-world, validated tasks collected from the LLVM project. We further develop LLVM-Gym, a scalable evaluation platform that automates issue reproduction, patch application, compiler building, and test execution. Using LLVM-Bench and LLVM-Gym, we conduct a comprehensive study of four representative LLMs, six retrieval configurations, and three agents. Our results show that current LLM-based issue resolution techniques remain limited on LLVM-Bench, with patch invalidity and build failures as the dominant failure modes. We further reveal a strong complementarity among different LLMs and agents, motivating LLVM-Ens, a lightweight ensemble approach that expands the patch space through integrating the patches generated by diverse techniques, filters incorrect and redundant candidates, and identifies the most promising solution. Our results show that LLVM-Ens achieves a resolution rate of up to 21.99%, further improving LLVM issue resolution.

</details>
